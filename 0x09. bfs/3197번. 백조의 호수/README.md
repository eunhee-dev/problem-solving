# 3197번. 백조의 호수

## 문제
https://www.acmicpc.net/problem/3197

---

## Key Points

1. [2573번. 빙산](../2573번.%20빙산/) 문제와 비슷하다.

      - 가장 먼저 떠오르는 풀이: 빙판 녹이기 => 출발점에서 bfs 돌려서 도착점에 도달하는지 보기 => 빙판 녹이기 ... 를 도착점에 도달할 때 까지 반복하는 것

      - 0 <= r, c <= 1500 이라 제한 시간 및 메모리 안에 통과가 불가능했다. O(rc* max(r, c))

2. Binary Search를 활용하여 최소 날짜 구하기: [풀이 - solve_bs.py](./solve_bs.py)

      - 날짜가 지난다고 빙산이 생기는 것이 아니기 때문에, 빙산이 다 녹는 최대 날짜와 0일을 기준으로 bs를 수행해서 만날 수 있는 최소 날짜를 찾는다.

      - 1번은 최악의 경우 max(r, c) 일에 대해 bfs를 돌려야할 수 있는데, bs를 사용하면 최악의 경우에도 log(max(r, c))만 bfs를 돌리면 된다.

3. bs를 안쓰더라도 조금 더 생각을 해보면, 빙판을 녹일 때 마다 백조의 출발점에서 도착점까지 bfs를 돌릴 필요가 없음을 알 수 있다. 

      - 백조가 빙판을 만나 더 이상 움직이지 못하는 상황까지는 저장해둘 수 있다. (bfs를 중간에 멈추기 - [16920번. 확장게임](../16920번.%20확장%20게임/) 문제와 유사)

      - 백조를 움직이게 못한 빙판은 물에 닿아있기 때문에, 다음에 녹을 빙판이다. 즉, 저장할 큐(`next_day_queue`)에는 다음 날에 백조가 움직일 수 있는 곳이다.  
      
      - 마찬가지로, 빙판을 녹이는 것에 대해서도 백조 (`swan_queue`) => 물(`water_queue`)로 생각하면 동일하게 시뮬레이션 할 수 있다.

      - 이 방법은 각 큐에 board가 한 번씩만 들어가기 때문에 O(rc)에 풀이 가능하다

      - 다만, 기존 queue와 별개로 next_day_queue에 다음 날 탐색할 좌표들을 저장하다보니, 기존 python 방식으로 구현하면 메모리 초과가 발생한다... (될 법 했는데.. C++은 됐을 것이다...)
            - 오기를 부려 `deque`나 `tuple`을 안쓰고 `array`, `bytearray` 등 최대한 물리적인 최적화를 해주니 통과하였다. (눈물의 메모리 절약 ...) [풀이 - solve_reuse.py](./solve_reuse.py)

      - 논리적으로 최적화가 가능할지 좀 더 고민해봤는데, MITM (Meet-in-the-Middle) 방식으로 출발점과 도착점을 동시에 큐에 넣고 만나는 순간을 체크해주면 메모리를 조금 더 아낄 수 있어 통과 가능하다. (중간에서 만날 경우 최대 절반 깊이를 줄일 수 있어 메모리 초과가 안난다.) [풀이 - solve_reuse_mitm.py](./solve_reuse_mitm.py)

4. bs를 떠올리지 못하면, (물리적 / 논리적) 메모리 최적화까지 고려해야 하는 아주 까다로운 문제였다.
